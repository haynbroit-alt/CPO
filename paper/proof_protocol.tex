\begin{abstract}
We present \textsc{Proof Protocol}, a system that transforms arbitrary
computational claims — including those generated by large language models
(LLMs) — into verifiable, reproducible, and cryptographically signed
execution artifacts called \emph{Computational Proof Objects} (CPOs).
A CPO bundles a natural-language claim, executable code, a controlled
sandbox environment, deterministic execution results, a SHA-256 content
hash, and an Ed25519 digital signature issued by the executing node.
Together, these fields form an immutable, auditable record of a computation.

Unlike zero-knowledge proof systems, \textsc{Proof Protocol} requires no
trusted setup, no cryptographic circuit compilation, and no global
consensus: verification consists of re-executing the same code under the
same constraints and comparing hashes.  We describe the architecture,
formalize the CPO data model, analyze the security properties, and
demonstrate that the approach is lightweight enough to deploy as a REST
API service today.  The system natively supports multiple isolated
execution environments — which we call \emph{worlds} — covering symbolic
mathematics, probabilistic inference, evolutionary computation, formal
verification, and multimodal data processing.
\end{abstract}

\section{Introduction}
\label{sec:intro}

Modern AI systems and automated computation pipelines share a common
weakness: their outputs are difficult to verify independently.  A language
model might produce a plausible-sounding numerical answer that is
arithmetically incorrect; a data-science pipeline might produce different
results on different machines; an autonomous agent might take actions whose
causal chain is impossible to reconstruct after the fact.

Three distinct communities experience this problem independently:

\paragraph{AI practitioners} observe that LLM outputs can \emph{hallucinate}
facts, including mathematical results, without any signal to the consumer
that the output is unreliable~\citep{ji2023survey}.

\paragraph{Scientists} struggle with \emph{computational reproducibility}:
a 2016 survey found that more than 70\% of researchers had failed to
reproduce another scientist's results, with software environment
differences as a major contributing factor~\citep{baker2016reproducibility}.

\paragraph{Auditors and compliance engineers} need \emph{auditability}:
the ability to prove \emph{post hoc} that a particular output was produced
by a particular computation running in a particular environment.

Existing solutions address each concern in isolation.  Reproducibility
tools (MLflow~\citep{mlflow}, Weights \& Biases) log hyperparameters but
not execution semantics.  Containerization (Docker, Nix) improves
environment reproducibility but does not bind execution to its output
cryptographically.  Verifiable computation systems (Truebit~\citep{truebit},
RISC Zero~\citep{risczero}) provide strong guarantees via zero-knowledge
proofs but impose circuit compilation overhead that is orders of magnitude
larger than the computation itself.

\textsc{Proof Protocol} occupies a different point in this design space:
a \emph{practical, lightweight verifiable execution layer} that cryptographically
binds computation inputs to outputs without zero-knowledge machinery or
blockchain consensus.  The key insight is that \emph{reproducible isolation
is a sufficient substitute for cryptographic universality} in many real-world
settings.

\section{The CPO Data Model}
\label{sec:model}

\begin{definition}[Computational Proof Object]
A \emph{Computational Proof Object} (CPO) is an immutable record
\[
  \mathcal{P} = (\textit{id},\; \textit{world},\; \textit{claim},\;
                  \textit{code},\; \textit{env},\; R,\; h,\; \sigma,\;
                  \textit{node},\; t)
\]
where:
\begin{itemize}
  \item $\textit{id} \in \{0,1\}^{128}$ is a globally unique identifier (UUIDv4).
  \item $\textit{world} \in \mathcal{W}$ selects an execution environment
        from a finite set of registered worlds (Section~\ref{sec:worlds}).
  \item $\textit{claim} \in \Sigma^*$ is a natural-language description of
        the hypothesis being tested.
  \item $\textit{code} \in \Sigma^*$ is deterministic, executable source
        code that operationalizes the claim.
  \item $\textit{env}$ is an execution specification (resource constraints,
        optional image override).
  \item $R = (\textit{stdout}, \textit{stderr}, \textit{exit\_code},
               \textit{runtime\_ms})$ is the observed execution result.
  \item $h = \mathrm{SHA\text{-}256}(\mathrm{canon}(\mathcal{P}
        \setminus \{h, \sigma\})) \in \{0,1\}^{256}$ is the content hash
        over the canonical serialization of all other fields.
  \item $\sigma = \mathrm{Sign}(\mathit{sk}_{\mathit{node}}, h)$ is an
        Ed25519 signature produced by the node's private key.
  \item $\textit{node} \in \{0,1\}^{256}$ is the public key of the signing node.
  \item $t \in \mathbb{Z}$ is the UTC timestamp of creation.
\end{itemize}
\end{definition}

The hash $h$ covers every field \emph{except} $h$ and $\sigma$ themselves,
so tampering with any single field invalidates both.  Crucially, $h$ is
computed over a \emph{canonical} serialization (Section~\ref{sec:crypto})
so that JSON key ordering or whitespace differences cannot produce false
inequalities.

The system exposes four primary REST endpoints:

\begin{itemize}
  \item \texttt{POST /prove} — accept $(world, claim, code)$, execute, produce
        and store a CPO; return its hash and signature.
  \item \texttt{POST /ask} — accept a natural-language \textit{question},
        generate \textit{code} via an LLM, then delegate to \texttt{/prove}.
  \item \texttt{GET /verify/\{hash\}} — retrieve a CPO by content hash,
        re-execute its code, and compare results.
  \item \texttt{GET /ledger} — browse the append-only ledger, optionally
        filtered by world.
\end{itemize}

\section{Execution Model}
\label{sec:execution}

\subsection{Sandbox Isolation}

Each execution is performed inside an ephemeral Docker container with the
following constraints enforced at the container-runtime level:

\begin{itemize}
  \item \textbf{Network isolation}: \texttt{network\_disabled=True} prevents
        any outbound or inbound traffic.
  \item \textbf{Read-only filesystem}: \texttt{read\_only=True} prevents
        persistent side effects.
  \item \textbf{Memory cap}: default 128\,MiB, configurable per CPO via
        \textit{execution\_spec.constraints}.
  \item \textbf{Execution timeout}: default 10\,s wall-clock time, also
        configurable.
  \item \textbf{Privilege suppression}: \texttt{no-new-privileges:true}
        prevents privilege escalation via \texttt{setuid} binaries.
\end{itemize}

These constraints together define what we call a \emph{bounded execution
environment}.

\subsection{Bounded Determinism}
\label{sec:determinism}

We adopt a pragmatic definition of determinism:

\begin{definition}[Bounded Determinism]
A computation $(code, env)$ is \emph{boundedly deterministic} if two
independent executions of $code$ in environments that are structurally
identical (same container image, same constraints, same input) produce
identical observable outputs $(\textit{stdout}, \textit{exit\_code})$.
\end{definition}

This is weaker than formal determinism: it does not account for
non-deterministic floating-point hardware behavior across CPU generations,
library version drift, or intentionally stochastic code.  For the use cases
targeted here — mathematical computation, symbolic reasoning, and
deterministic algorithms — bounded determinism holds in practice.

\section{Cryptographic Layer}
\label{sec:crypto}

\subsection{Canonical Serialization}

To ensure that $h$ is stable across JSON implementations, we serialize the
CPO (excluding $h$ and $\sigma$) using an RFC~8785-inspired canonical form:
keys are sorted lexicographically at every nesting level, no extra
whitespace is inserted, and Unicode characters are preserved without
escaping.  Formally, for an object $O$:

\[
  \mathrm{canon}(O) \;=\; \texttt{JSON.dumps}(O,\;
    \texttt{sort\_keys=True},\;
    \texttt{separators=(",",":")},\;
    \texttt{ensure\_ascii=False})
\]

\subsection{Content Hash}

\begin{equation}
  h \;=\; \mathrm{SHA\text{-}256}\!\left(\mathrm{canon}
          \left(\mathcal{P} \setminus \{h, \sigma, \textit{node}\}\right)
          \;\text{encoded as UTF-8}\right)
  \label{eq:hash}
\end{equation}

\subsection{Node Identity and Signature}

Each node generates an Ed25519 keypair on first startup and persists the
private key in a PEM file.  The node's identity is:
\[
  \textit{node\_id} \;=\; \mathrm{SHA\text{-}256}(\textit{pub\_bytes})
                    \;[\,{:}16\,]\; \text{(hex prefix)}
\]
The signature is:
\[
  \sigma \;=\; \mathrm{Ed25519.Sign}(\mathit{sk}, \; h \cdot \text{UTF-8})
\]
where $h$ is the hex-encoded hash string of Equation~\eqref{eq:hash}.

\begin{property}[Tamper Evidence]
For any CPO $\mathcal{P}$, modifying any field $f \notin \{h, \sigma\}$
produces $h' \neq h$, which causes $\mathrm{Verify}(\textit{node},\,
h',\,\sigma) = \bot$.
\end{property}

\section{World Abstraction}
\label{sec:worlds}

A \emph{world} is a named, versioned Docker image that provides a specific
runtime environment.  Each world defines a domain of computation with its
own library ecosystem and behavioral assumptions.

\centering

\label{tab:worlds}
\begin{tabular}{llll}
\toprule
\textbf{World} & \textbf{Primary domain} & \textbf{Key libraries} & \textbf{Verification mode} \\
\midrule
\texttt{llm}         & General Python / LLM tools    & stdlib              & re-execution \\
\texttt{symbolic}    & Symbolic mathematics          & SymPy, Z3           & re-execution \\
\texttt{bayesian}    & Probabilistic inference       & NumPyro, PyMC       & bounded re-exec \\
\texttt{evolutionary}& Agent-based simulation        & DEAP, Mesa          & bounded re-exec \\
\texttt{formal}      & Logic and SMT solving         & Z3, py-aiger        & re-execution \\
\texttt{multimodal}  & Vision and audio              & PyTorch CPU, Pillow & re-execution \\
\texttt{neuro}       & Neural pipeline tooling       & DSPy, LangChain     & bounded re-exec \\
\bottomrule
\end{tabular}
\end{table}

Worlds with \emph{bounded re-execution} (stochastic by nature) can still be
verified for signature validity and environment integrity, even if exact
output reproduction is not guaranteed without fixing the random seed in the
submitted code.

\section{Verification Procedure}
\label{sec:verification}

Given a content hash $h$, the verification procedure $\mathrm{Verify}$ is:

\begin{enumerate}
  \item \textbf{Retrieve}: look up $\mathcal{P}$ in the ledger by $h$.
  \item \textbf{Signature check}:
        $\mathrm{Ed25519.Verify}(\mathcal{P}.\textit{node},\; h,\;
        \mathcal{P}.\sigma) \stackrel{?}{=} \top$
  \item \textbf{Re-execute}: run $\mathcal{P}.\textit{code}$ in world
        $\mathcal{P}.\textit{world}$ with the same constraints to obtain
        fresh result $R'$.
  \item \textbf{Output match}:
        $R'.\textit{stdout}.\mathrm{strip}() \stackrel{?}{=}
        \mathcal{P}.R.\textit{stdout}.\mathrm{strip}()$
  \item \textbf{Exit code match}:
        $R'.\textit{exit\_code} \stackrel{?}{=} \mathcal{P}.R.\textit{exit\_code}$
\end{enumerate}

The CPO is considered \emph{verified} if and only if all five checks pass.
Steps 1--2 require no re-execution and provide integrity and authenticity
independently of determinism.

\section{Security Analysis}
\label{sec:security}

\subsection{Threat Model}

We consider the following adversaries:

\paragraph{Passive ledger reader.}
An adversary who reads CPOs from the ledger but cannot modify them.  The
append-only structure and cryptographic hashing prevent undetected
modification.

\paragraph{Malicious code submitter.}
An adversary who submits code intended to escape the sandbox or affect the
host.  Network isolation, read-only filesystem, seccomp profiles, and no-new-privileges collectively reduce the attack surface.

\paragraph{Compromised node.}
An adversary who controls the node's private key.  Signatures from a
compromised key remain verifiable, but the node identity allows attribution.
Key rotation invalidates future attestations but not past ones (since past
CPOs carry the correct public key inline).

\subsection{Non-Goals}

\textsc{Proof Protocol} explicitly does not provide:
\begin{itemize}
  \item \textbf{Semantic correctness}: a signed CPO proves that code
        \emph{ran}, not that the code \emph{correctly answers} the claim.
  \item \textbf{Global consensus}: there is no mechanism to prevent two
        nodes from producing conflicting CPOs for the same claim.
  \item \textbf{Kernel-level sandbox security}: Docker-based isolation
        does not protect against kernel exploits.  Firecracker
        microVMs~\citep{firecracker} or gVisor would strengthen this
        boundary at higher operational cost.
  \item \textbf{Confidentiality}: CPOs contain the executed code and its
        output in plaintext.
\end{itemize}

\section{Limitations}
\label{sec:limitations}

\paragraph{Determinism scope.}
Stochastic worlds (Bayesian, Evolutionary) cannot guarantee identical
outputs across re-executions without explicit random seed management by the
code author.

\paragraph{Container security.}
Docker shares the host kernel.  A kernel-level exploit may escape the
sandbox.  Production deployments should use microVM isolation.

\paragraph{LLM code quality.}
The \texttt{/ask} endpoint relies on an LLM to generate correct code.  A
hallucinated or incorrect code snippet produces a signed CPO for a wrong
answer.  The signature attests to \emph{execution}, not to
\emph{correctness}.

\paragraph{Ledger scalability.}
The current JSONL ledger is append-only and single-file.  At scale, this
requires partitioning by world or time range, or migration to a structured
event store.

\paragraph{No cross-node consensus.}
Each node is independently trusted.  A network of nodes providing
quorum-based verification is a planned extension, not a current guarantee.

\section{Related Work}
\label{sec:related}

\paragraph{Reproducible research.}
MLflow~\citep{mlflow} and Weights \& Biases record hyperparameters and
artifacts but do not cryptographically bind execution environments to
outputs.  Nix~\citep{nix} and Guix provide reproducible builds but are
scoped to software delivery, not runtime execution.

\paragraph{Verifiable computation.}
Truebit~\citep{truebit} introduces interactive proofs for on-chain
verification of off-chain computation.  RISC Zero~\citep{risczero} and
StarkWare compile arbitrary programs to ZK circuits, providing
unconditional soundness at the cost of circuit compilation time.
Cartesi~\citep{cartesi} runs Linux applications in a reproducible
RISC-V machine to generate verifiable computation traces.
\textsc{Proof Protocol} trades ZK soundness for simplicity: verification
is re-execution rather than proof checking.

\paragraph{Trusted Execution Environments.}
Intel SGX and AMD SEV provide hardware-level attestation of code running
inside enclaves~\citep{sgx}.  These offer stronger isolation but require
specific hardware and attestation infrastructure.
\textsc{Proof Protocol} is hardware-agnostic.

\paragraph{Secure sandboxing.}
Firecracker~\citep{firecracker} provides lightweight microVM isolation
with sub-125\,ms boot times, used in AWS Lambda and Fargate.
gVisor~\citep{gvisor} intercepts syscalls in user space.
Both are compatible with \textsc{Proof Protocol}'s execution layer and
represent a natural upgrade path from Docker.

\paragraph{LLM grounding.}
Tool-augmented language models~\citep{toolformer, react} address
hallucination by grounding LLM outputs in external tool calls.
\textsc{Proof Protocol}'s \texttt{/ask} endpoint can be viewed as an
instance of this paradigm where the tool is a verifiable code executor.

\section{Conclusion}
\label{sec:conclusion}

We have presented \textsc{Proof Protocol}, a lightweight verifiable
execution layer that transforms computational claims into signed, auditable,
and reproducible artifacts.  The system is grounded in three principles:

\begin{enumerate}
  \item \textbf{Isolation implies reproducibility}: bounded execution
        environments make re-verification possible without ZK machinery.
  \item \textbf{Canonical hashing implies integrity}: any tampering with
        a CPO is detectable by recomputing $h$.
  \item \textbf{Signed attestation implies non-repudiation}: node identity
        is cryptographically bound to every CPO it issues.
\end{enumerate}

Together, these properties position \textsc{Proof Protocol} as a practical
trust layer for AI pipelines, scientific computation, and autonomous agents.

\paragraph{Availability.}
A reference implementation in Python (FastAPI, Ed25519, Docker) is available
at \url{https://github.com/haynbroit-alt/CPO}.
